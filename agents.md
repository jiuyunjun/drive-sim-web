# agents.md

## 项目概况

- 项目名称：`drive-sim-web`
- 当前形态：纯前端静态页面，核心文件为 `index.html`、`app.js`、`style.css`
- 运行方式：通过本地静态文件服务器启动，不要直接双击 `index.html`
- 3D 依赖来源：`index.html` 中通过 `importmap` 从 `unpkg` 加载 `three` 与附加模块

## 协作约束

- 所有新增或修改文件统一使用 UTF-8 编码
- 每次独立修改单独提交，提交信息要能说明改动目的
- 不要无理由重命名现有入口文件
- 未经确认，不要把资源改成需要后端服务才能运行的形态

## UTF-8 读写要求

- 读取和保存文本文件时，统一按 UTF-8 处理，不要改成 GBK、ANSI、Shift_JIS 或 UTF-16
- 终端里看到乱码时，先区分“控制台显示问题”和“文件真实编码问题”，不要仅凭终端输出判断文件损坏
- 修改文件前后，优先使用能明确按 UTF-8 读写的工具和流程
- 如果需要做编码校验，优先做字节级验证，不要只看肉眼显示结果

### 推荐做法

- 编辑时优先使用支持 UTF-8 的编辑器或补丁式修改流程
- 在 PowerShell 中查看文件内容时，如果显示异常，先不要直接重存文件
- 需要验证时，可用 Node.js 按 UTF-8 读取并做 round-trip 校验：文件字节 -> UTF-8 字符串 -> UTF-8 字节，前后完全一致即可视为编码正常
- 若文件内容需要批量转换，先备份，再明确指定源编码和目标编码，避免误转码

### 处理原则

- 未经确认，不要因为终端乱码就批量重写仓库文件
- 如果文件本身已是 UTF-8，只修正查看方式，不重复转码
- 新增文档默认使用 UTF-8 无 BOM，除非工具链明确要求 BOM

## 本地开发

1. 在仓库根目录启动静态服务器，例如：

```bash
python -m http.server 8000
```

2. 浏览器打开：

```text
http://localhost:8000/
```

## 代码结构

- `index.html`：页面结构、UI 面板、`importmap`
- `app.js`：Three.js 场景、车辆控制、音效、后视镜、小地图、设置持久化
- `style.css`：界面样式
- `map.png`、`map2.webp`：示例地图资源
- `start.bat`：本地启动辅助脚本

## GCP 预备要求

- 当前项目适合优先部署到 GCP 的静态托管路径，例如 Cloud Storage + Cloud CDN，或 Firebase Hosting
- 如果后续要上 Cloud Run，应先补充明确的静态服务入口，例如 `nginx` 或轻量 Node.js server
- 若继续依赖 `unpkg`，上线前要确认外网访问策略；更稳妥的做法是后续将前端依赖改为本地锁定版本
- 部署前应补充：
  - 生产环境缓存策略
  - 资源版本管理
  - 基础可用性检查
  - 控制台报错检查
